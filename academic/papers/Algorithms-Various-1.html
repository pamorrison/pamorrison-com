Computer Science Algorithms: Problems and Solutions 1

 

Explain how to solve the following two problems using heaps.

a) First, give an O (n log k) algorithm to merge k sorted lists with n total elements into one sorted list.

         Algorithm:

1. Assign to each sorted list a marker such that there are k values (list markers) to pair with keys (list elements) from each list. (k steps)

2. For each of the k sorted lists, remove the first/least element, create a key/value pair using the marker from step 1, and put it into a min heap structure (k steps), Q (stored as a list, so that each insertion occurs at the end), allowing every insertion to obey the heap ordering rule and each item to “bubble up” if necessary. At the end of this initial process, there exists a min heap of size k (~k*log k additional steps).

3. Create an empty list L in which to store the list elements obtained and removed from Q (constant).

4. While Q is non–empty: Remove Q’s root (extract min, ~log k), store the value (original list marker) in a temporary variable t, and insert the key (original list element) at the end of L (constant). Afterward, derive the next key/value pair from the next element in list t, if list t is non–empty, and insert it into the heap such that the new root’s “key” becomes the least element out of all elements in all lists (~log k). Skip the insertion if list t is empty. Start over at step 4 (total of n – k steps).

5. When Q becomes empty, we are left with L, a sorted list of all n elements.

Explanation:

With a little bit more prep work than just relying on a min heap to give us an ordering, specifically by “tagging” elements with an original list identity, we have eliminated the need to hunt for a specific list element to insert into the heap, instead allowing the removed root to always tell us where to go get a new element. Summing the bounds across all steps we have:

 O(k) + O(k) + O(klog k) + O(1) + O(2(n – k)log k) = O(k + k + 1 + ­­­ klog k +  2(n – k)log k)

≈ O(n log k).

In this case, we can treat k as a constant for determining the running time, because given a particular n, for increasing k, the running time of steps 1 and 2 increase while step 4 decreases, so we end up with a running time that is primarily determined by the total number of elements, rather than how many lists they’re divided into.

b) Second, say that a list of numbers is k – close to sorted if each number in the list is less than k positions from its actual place in the sorted order. Give an O (n log k ) algorithm for sorting a list of n numbers that is k – close to sorted.

Algorithm:

1. Begin by removing k+1 elements (k+1 steps) from the list and inserting them into a min–heap structure (k+1 steps), Q, allowing the heap properties to “sort” these elements (~log k per element).

2. Create an empty list L for output (constant).

3. While Q is non–empty: Remove an element from Q (extract min, ~log k) and place it at the end of L. Insert the next element of the original list into the heap (~log k). Repeat step 3 (total of n – k – 1 steps).

4. After Q is empty, we have L, a sorted list of n elements.

Explanation:

The sorting comes almost entirely from simply using a min–heap structure, allowing it to produce least elements. By starting with a valid min–heap at the beginning with k + 1 elements, the first iteration of the heap is guaranteed to have at least the first k elements correctly sorted, since the first element in the final sorted list is only k positions away from its correct position. Summing the bounds across all algorithm steps, we have

O((2k + 2)log k) + O(1) + O(2(n – k – 1)log k)) = O((2k + 2)log k + 1 + 2(n – k – 1)log k))

≈ O(n log k).

Similar to part (a), the latter part of the algorithm is decreased in running time by increasing k, whereas the former part is increased, so k can be treated as a constant, so the overall running time is primarily dependent on the total number of elements being sorted.

Design an efficient algorithm to find the longest path in a directed acyclic graph.

We consider a DAG called G = {V,E } with vertex list V and edge list E that contains edge objects with source and destination, as well as weight if given, where |V| = n and |E| = m. We also consider a requested source v’.

Algorithm:

1. Create a 2–d adjacency list representation for G called A: for each edge e in E, with source e1 , destination e2 , and weight w (if it exists, assigning w = 1 otherwise), creating the sub–list for e1 if necessary, store a duple object containing e2 and w in e1‘s sub–list A[e1] (~n + m steps).

2. Create a topological sorting of G in a stack called S: for each node A[i], perform a recursive DFS on its neighbors using the information in row A[i], and push a copy of A[i] onto S once the DFS completes. For every node pushed onto S in this way, the DFS can be skipped, so attach a “pushed t/f” value to each A[i] so that it is only visited once (~n steps).

3. Initialize a distances list D of length n (~n steps) such that every value in D is a default “lower than the lowest weight” value (such as –1000 or –∞) except the index of v’, which is instead initialized to 0 (~n steps). Then for every node referenced by S, starting at the top, process every neighbor via indexing A and updating every matching distance in D by comparing the current neighbor distance and the current source distance: if D[neighbor] < D[current] + weight(current,neighbor), then D[neighbor] becomes this sum (~n + m steps).

4. After step 3, we are left with D, a list of the longest distances from v’ to every node (ranging from either the default minimum described earlier or 0 to the longest distance), so the length of the longest path in the graph from v’ to any node is simply the max of D.

Explanation:

This algorithm is designed with more prep work than would normally be necessary for a shortest–path algorithm, but as explained in step 4, it provides a complete list of longest path lengths by modifying the cost function of the standard shortest path algorithm for DAGs. There are no inner loops that blow up the run time, and every set of steps is bounded either by n + m or n, so after summing all of the steps, the run time is bounded above by O(n + m), linear time.

You’re helping some security analysts monitor a collection of networked computers... tracking the spread of an online virus. There are n computers in the system, call them C1 , C2 , ... , Cn. You are given a trace indicating the times at which pairs of computers communicated. The trace consists of m triples (Ci , Cj ,  tk) that indicate that Ci and Cj communicated at time tk . At this time, a virus could have spread from Ci to Cj , or from Cj to Ci.

We assume that the trace holds the triples sorted in order by time. For simplicity, we assume that each pair of computers communicates at most once over the time of the trace. Also, it is possible to have pairs (Cs ,Cj , tk) and (Ct , Cj , tk); this would indicate that Cj opened connections to both Cs and Ct at time tk , allowing a virus to spread in any way among the 3 machines. There’s nothing special about 3 here – there might be 4 or 10 or however many connections open at the same time, allowing a virus to spread in any way among all of them.

We would like to answer questions of the following form: if a virus was introduced to Ci at time x, could it have spread to Cj at time y? That is, is there a sequence of communications that could have led from the virus moving from Ci to Cj?

Design an algorithm that, given as input a collection of (sorted) trace data and a virus query, gives a yes/no answer for the query. The algorithm should run in time O (m + n).

Algorithm:

1. Scan the trace data and discard all trace elements until the first timestamp at or after the virus introduction time is encountered. Once the time at or after the virus introduction time is encountered, save it in a variable t (m – c + 1 steps).

2. Create and initialize an array A that can be indexed by the computer number and mark the initially infected computer and mark all others as uninfected (n steps).

3. While the time encountered in the trace data matches t, treat the two computers in each trace element as two nodes and create a graph G (adjacency list) using every trace element that matches the current time. Once you encounter a new time, save it in t and stop accepting input for the moment (n + m + 1 steps).

4. Perform a BFS on G repeatedly to find and create connected components. For each component, saved in a simple array, check each computer (node) via constant time access to A whether any computer in that component is infected, and if so, update A to reflect that every computer in that component is now infected. This check is performed for every computer, which are themselves present in only one component, regardless of the number of components (m – di steps).

5. Repeat steps 3 and 4 until one of 3 outcomes: the originally requested computer to find a path to has been positively identified in step 4 as infected OR the trace input runs out OR the time t has reached or exceeded the time limit originally given. Thus, either the 1st outcome occurs, resulting in a “yes” or one of the other 2 outcomes occur, resulting in a “no” ( m – (m – c) = c steps ).

Explanation:

The algorithm is designed to reach an absolute outcome, so it will always produce the requested yes/no answer as explained in step 5. The steps involving creating the infection array A and updating it are bound by n, since a computer can only be infected once. The steps involving creating graphs from the trace data and creating connected components are bound by m, since there are a constant number of steps performed for each trace element (graph edge), and the total number of trace elements is m. All other steps performed are either constant time or directly based on either the number of computers or the number of trace elements, and there are no inner loops that would cause a proportional increase in run time, so this is a linear time algorithm bounded above by O(m + n).

Giles has asked Buffy to optimize her procedure for nighttime patrol of Sunnydale. (This takes place sometime in Season 2.) Giles points out that proper slaying technique would allow Buffy to traverse all of the streets of Sunnydale in such a way that she would walk on each side of each block, exactly once. Buffy now has slayer homework: how can it be done?

Assumptions about Sunnydale: we assume that Sunnydale is not some kind of multi–level city and that any elevations can be “flattened out” into a 2–dimensional road map that can be abstracted as an unweighted, undirected, connected graph. The roads in Sunnydale are the edges of the graph and the intersections (4–way and 3–way) and cul–de–sacs are the nodes of the graph.

Thus, it becomes Buffy’s job to traverse the entire graph, visiting every edge exactly once so that she can take an exactly reversed path to travel the “other side” for the second time. For any unweighted, undirected, connected graph, there certainly exists a path to every node exactly once because of the existence and operation of the DFS traversal algorithm, so to start with, we can perform this search with dynamic programming starting at any node to create most of the journey. While visiting every node, treat the recursive call as walking “one side” on a street, and treat the eventual return as walking “the other side” of the street. This provides the bulk of a forward path and a reverse path to traverse every edge. The only issue is that in the normal operation of DFS with dynamic programming, the search function is only called on nodes that have not been visited, so there are edges that would typically be skipped so that the nodes themselves are only visited once. However, in our real–world scenario, we can simply instruct Buffy that, for every node (in these cases, they would be intersections) that has already been visited, she needs to walk to it and immediately return such that although there may be intersections that will be visited multiple times, every street is walked down in both directions exactly once. In an abstract sense, this creates a path that includes all of the edges that are normally excluded when using DFS to create the DFS tree via the standard creation of an adjacency list or matrix.

Suppose instead that we are concerned with the length of the longest edge between the source and every node in a graph. That is, the “bottleneck” of a path is defined to be the length of the longest edge in the path. Design an efficient algorithm to solve this single source smallest bottleneck problem; i.e. find the paths from a source to every other node such that each path has the smallest possible bottleneck.

Algorithm (modified Djikstra’s):

1. Assume we have an adjacency list representation called A that includes a leftmost column to be used for “visited/unvisited”. Also create a 1–dimensional array B that is parallel to A and assign a default “high” value (such as 10,000 or infinity) for every node except for the requested source node, which should be marked as 0. The values in B represent the path cost, which in this case is the smallest maximum weight for all paths from the source node. Also create an empty min–heap Q to use as a priority queue that accepts objects that use max weight as a key and node index as a value and insert the source node with weight zero into Q. Set current node to the source node.

2. Perform extract_min (and heapify) on Q and consider this the current node. Use the current node to access A to find its unvisited neighbors. For each neighbor, calculate the following: min (lowest max distance from B for this neighbor, distance from the current node to this neighbor). This minimum value replaces the value in B, so an update occurs if the min is lower than what is currently held in B. Create a Q object from this minimum value and the neighbor node and insert it into Q (and heapify).

3. When all of the unvisited neighbors of the current node have been processed, mark the current node as visited in A.

4. Repeat steps 2 and 3 until Q is empty.

5. We are left with B, an array that is parallel to A and thus for each node A[i], then B[i] is the requested shortest bottleneck for any path from A[source] to A[i].

Explanation:

This algorithm works because it explores nodes with the same style of exploration as Djikstra’s algorithm with only a change in the prioritization of which node to choose as “current” (decided by the heap) and how that prioritization is calculated (smallest bottleneck instead of lowest path cost). Since all the implementation steps are identical to Djikstra’s with the exception of the cost function (which is constant–time, like Djikstra’s), the run time is essentially identical as well, so excluding the time it takes to create the graph, it is bounded above by O(mlog n).

Consider the shortest paths problem in the special case where all edge costs are non–negative integers. Describe a modification of Dijkstra’s algorithm that works in time O( |E | + |V |m ), where m is the maximum cost of any edge in the graph.

Since the edge costs are all non–negative integers, the structure to store nodes and the structure to store distances can be combined into a list of lists (buckets). So then to implement this change, instead of creating a min–heap in which to store nodes, a standard integer–indexed list, say it’s called L, can be created with size m + 1. Then, whenever a minimum cost is calculated for a neighbor to the current node, the neighbor’s identifier can be inserted at the end of a linked list at the position in L that exactly matches that minimum cost value, essentially using the cost as an index for constant–time insertion, and if insertion had already occurred, determining whether a node should be moved to a lower bucket and which bucket to move it to is also constant–time. This also creates the advantage that, provided we keep track of where in L our last node was removed from, we can simply start scanning L from that position to find the next non–empty “bucket” to retrieve our next current node, which replaces the “extract_min” operation, so the total number of retrieval steps is bounded by the number of “buckets” (m + 1). Keeping in mind that this scheme only allows nodes to either be inserted or moved from greater buckets to lower buckets, then the run time can be calculated as follows: there is some constant number of steps for every edge and in the worst (impossible) case, each node will take some constant number of steps to move into a bucket, then be moved left by a number proportional to m, then finally discarded. Therefore, the run time for the improved algorithm is bounded by a) the total number of edges + b) the number of vertices multiplied by the maximum cost of any edge, giving us the desired O(|E| + |V|m ).
